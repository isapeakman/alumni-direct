# 问答模式规则 - 先方案后实现

## 📋 核心原则

**每次用户提问时，AI 必须先提供实现方案，待用户确认后再进行代码实现。**

## 🎯 工作流程

### 第一步：问题分析与方案设计

当用户提出需求或问题时，AI 应该：

1. **理解问题本质**
    - 分析用户需求的核心目标
    - 识别关键约束条件
    - 明确预期结果

2. **提供多种解决方案**
    - 至少提供 2-3 种可行方案
    - 每种方案包含：
        - ✅ 实现思路概述
        - ✅ 优点分析
        - ✅ 缺点分析
        - ✅ 适用场景
        - ✅ 实现复杂度评估（⭐数量）

3. **方案对比与建议**
    - 使用表格对比各方案
    - 给出明确的推荐建议
    - 说明推荐理由

4. **等待用户确认**
    - 明确询问用户选择哪个方案
    - 不要主动开始编写代码

### 第二步：代码实现

**只有在用户明确确认方案后**，AI 才能：

1. 按照用户选择的方案进行实现
2. 提供完整的代码实现
3. 说明关键实现细节
4. 提供测试建议

## ⚠️ 注意事项

1. **禁止直接实现代码**
    - 未经用户确认，不得直接编写代码
    - 即使用户说"直接实现"，也应先简要说明方案

2. **方案要具体可执行**
    - 避免过于抽象的描述
    - 包含关键技术点和实现步骤

3. **考虑多种维度**
    - 技术可行性
    - 性能影响
    - 用户体验
    - 维护成本
    - 扩展性

4. **特殊场景处理**
    - 如果是紧急 bug 修复，可以提供快速方案 + 长期方案
    - 如果是优化类需求，重点对比性能提升和实现成本

## ✅ 示例对话

**用户**：前端简历解析页面存在后端还没返回，直接就解析失败的问题

**AI**：

```
## 🔍 问题分析
前端在调用后端简历解析接口时，由于后端处理时间较长（OCR识别 + GLM API调用通常需要几秒），前端可能在请求超时前就判断为失败。

## 💡 解决方案

### 方案1：增加前端请求超时时间 ⭐
- **实现方式**：在前端 request.js 中针对简历解析接口设置更长的超时时间（如 30-60 秒）
- **优点**：简单直接，快速解决问题
- **缺点**：治标不治本，用户体验仍有等待焦虑
- **适用场景**：快速修复，临时方案

### 方案2：添加加载状态和进度提示 ⭐⭐ 推荐
- **实现方式**：前端显示"解析中..."的loading状态，显示分阶段进度提示
- **优点**：提升用户体验，让用户知道系统在正常工作
- **缺点**：仍需等待较长时间
- **适用场景**：改善用户体验的标准做法

### 方案3：异步任务 + 轮询查询 ⭐⭐⭐ 最佳体验
- **实现方式**：前端上传文件后，后端立即返回任务ID，前端每隔2-3秒轮询查询任务状态
- **优点**：不会超时，可以显示实时进度，支持后台处理
- **缺点**：实现复杂度较高，需要改造前后端
- **适用场景**：追求最佳用户体验，长期方案

## 📊 方案对比
...

## ❓ 请确认
你希望采用哪个方案？
```

**用户**：1. 先使用方案3实现

**AI**：[开始实现方案3的代码]

## 🔄 持续优化

此规则应根据实际使用情况不断优化，确保 AI 与用户的协作效率最大化。
